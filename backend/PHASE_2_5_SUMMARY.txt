LexAI Backend - Phase 2.5 Implementation Summary
==================================================

OBJECTIVE:
Implement the Document Upload & Management module that enables users to upload legal documents (PDF/DOCX), extract text content, store metadata and extracted text in MongoDB, and manage documents through CRUD operations. This module provides the foundation for all AI features (summary, risk analysis, chat, financial extraction, translation, report generation) by making extracted document text available for AI processing.

ALIGNMENT WITH PRD:
This implementation follows the PRD's "Upload Legal Document" feature specification exactly:
- Supports PDF and DOCX formats
- Enforces 25 MB maximum file size (configurable)
- Extracts text from uploaded documents
- Stores original text in MongoDB
- Makes documents ready for AI analysis

The implementation establishes the data flow foundation described in the PRD:
  Upload PDF → PyMuPDF → Extract Text → MongoDB → (Future: Semantic Kernel → AI Foundry)

IMPLEMENTATION OVERVIEW:

The Document Management module consists of four architectural layers:

1. API Layer (Routers)
   - Defines HTTP endpoints for document operations
   - Handles request validation and response formatting
   - Delegates business logic to service layer

2. Service Layer
   - Implements business logic for document operations
   - Coordinates between validation, storage, extraction, and database operations
   - Manages file system operations and cleanup on errors

3. Repository Layer
   - Encapsulates all MongoDB database operations
   - Provides clean abstraction over database queries
   - Handles ObjectId conversions and projections

4. Utility Layer
   - Text extraction from PDF and DOCX files
   - Validation logic
   - Error handling

API ENDPOINTS IMPLEMENTED:

1. POST /documents/upload
   - Accepts multipart/form-data file upload
   - Validates file type, size, and format
   - Saves file to disk with unique filename
   - Extracts text content
   - Stores metadata and text in MongoDB
   - Returns document ID and upload details

2. GET /documents/
   - Lists all uploaded documents with pagination
   - Supports limit and skip query parameters
   - Returns metadata only (excludes document text for performance)
   - Returns total document count

3. GET /documents/{document_id}
   - Retrieves a specific document by ID
   - Includes full metadata and extracted text by default
   - Supports optional include_text parameter to exclude text

4. DELETE /documents/{document_id}
   - Deletes document record from MongoDB
   - Deletes physical file from disk
   - Returns success confirmation

DOCUMENT UPLOAD WORKFLOW:

Step 1: Request Reception
- FastAPI receives multipart/form-data request
- File is parsed and wrapped in UploadFile object

Step 2: Validation (DocumentService._validate_file)
- Check file exists and has filename
- Extract file extension from filename
- Validate extension against supported types (pdf, docx)
- Validate MIME type matches file extension
- Raises ValidationError if any check fails

Step 3: File Storage (DocumentService._save_file)
- Generate unique filename using UUID
- Read file content into memory
- Validate file size against MAX_FILE_SIZE_MB
- Check for empty files
- Save file to uploads/ directory with unique name
- Returns stored filename and file size

Step 4: Text Extraction (DocumentService._extract_text)
- Route to appropriate extractor based on file type
- PDF: Use PyMuPDF to extract text from all pages
- DOCX: Use python-docx to extract text from all paragraphs
- Validate extracted text is not empty
- Returns full extracted text content

Step 5: Database Storage (DocumentRepository.create_document)
- Create DocumentCreate model with all metadata
- Add uploadedAt timestamp
- Insert document record into MongoDB documents collection
- Returns generated document ID

Step 6: Response
- Return UploadResponse with document ID, filename, file size, and text length
- Client receives confirmation and document ID for future operations

Error Handling:
- If any step fails after file save, automatically delete the saved file
- This ensures no orphaned files remain on disk
- All errors are logged with context

FILE VALIDATION PROCESS:

Extension Validation:
- Extract extension from filename using Path().suffix
- Normalize to lowercase and strip leading dot
- Compare against SUPPORTED_FILE_TYPES from configuration
- Reject if not in allowed list

MIME Type Validation:
- Check Content-Type header from upload
- PDF must be: application/pdf
- DOCX must be: application/vnd.openxmlformats-officedocument.wordprocessingml.document
- Also accept application/octet-stream for DOCX (browser compatibility)
- Log warning if MIME type doesn't match but continue (some browsers send incorrect types)

Size Validation:
- Read file content to determine actual size
- Compare against MAX_FILE_SIZE_MB * 1024 * 1024
- Reject files exceeding limit with clear error message

Content Validation:
- Check for zero-byte files
- Verify extracted text is not empty
- Detect corrupted files during extraction

TEXT EXTRACTION PROCESS:

PDF Extraction (extract_text_from_pdf):
- Uses PyMuPDF (fitz) library
- Opens PDF document from file path
- Iterates through all pages
- Extracts text from each page using page.get_text()
- Combines page text with double newline separators
- Validates result is not empty
- Closes document properly
- Handles corrupted PDFs with FileDataError exception

DOCX Extraction (extract_text_from_docx):
- Uses python-docx library
- Opens DOCX document from file path
- Iterates through all paragraphs
- Extracts text from each paragraph
- Filters out empty paragraphs
- Combines paragraph text with double newline separators
- Validates result is not empty
- Handles corrupted DOCX files gracefully

Error Handling:
- Corrupted files raise ValidationError with clear message
- Empty documents raise ValidationError
- All extraction errors are logged with context
- Failed extractions trigger file cleanup

MONGODB STORAGE FLOW:

Collection: documents

Document Schema:
{
  _id: ObjectId,
  fileName: string,              (Original filename)
  storedFileName: string,         (UUID-based filename on disk)
  fileType: string,               (pdf or docx)
  fileSize: integer,              (Bytes)
  uploadedAt: datetime,           (UTC timestamp)
  documentText: string,           (Extracted text content)
  language: string,               (Default: "english")
  userId: string (optional)       (Placeholder for future auth)
}

Storage Process:
1. DocumentCreate model validates input data
2. Convert model to dictionary
3. Add uploadedAt timestamp (UTC)
4. Insert into documents collection
5. Extract and return inserted_id as string
6. Log successful creation

Retrieval Optimizations:
- List operations exclude documentText field (projection)
- This reduces network transfer and improves performance
- Full text only retrieved when explicitly requested
- ObjectId automatically converted to string for JSON serialization

REPOSITORY LAYER RESPONSIBILITIES:

DocumentRepository provides clean database abstraction:

create_document(document: DocumentCreate) -> str
- Inserts new document into MongoDB
- Adds uploadedAt timestamp
- Returns document ID as string
- Raises DatabaseError on failure

get_document_by_id(document_id: str, include_text: bool) -> Optional[dict]
- Validates ObjectId format
- Uses projection to conditionally exclude documentText
- Converts ObjectId to string for JSON compatibility
- Returns None if document not found
- Raises DatabaseError on database errors

list_documents(limit: int, skip: int) -> tuple[List[dict], int]
- Queries with documentText projection excluded
- Sorts by uploadedAt descending (newest first)
- Applies pagination with skip and limit
- Converts all ObjectIds to strings
- Returns tuple of (documents, total_count)
- Raises DatabaseError on failure

delete_document(document_id: str) -> bool
- Validates ObjectId format
- Deletes document from collection
- Checks deletion count to verify document existed
- Raises NotFoundError if document doesn't exist
- Raises DatabaseError on database errors

Design Pattern:
- Repository isolates database operations from business logic
- Services depend on repositories, not direct database access
- Enables easier testing and potential database migration
- Follows Clean Architecture dependency rule

SERVICE LAYER RESPONSIBILITIES:

DocumentService orchestrates document operations:

Initialization:
- Receives MongoDB database instance via dependency injection
- Creates DocumentRepository instance
- Loads configuration settings
- Creates uploads directory if it doesn't exist

upload_document(file: UploadFile, user_id: str) -> UploadResponse:
- Coordinates complete upload workflow
- Validates file format and size
- Saves file to disk with unique name
- Extracts text content
- Creates database record
- Implements cleanup on error
- Returns detailed upload response

get_document(document_id: str, include_text: bool):
- Retrieves document from repository
- Returns DocumentResponse or DocumentWithText based on include_text
- Raises NotFoundError if document doesn't exist

list_documents(limit: int, skip: int) -> DocumentListResponse:
- Retrieves paginated document list from repository
- Converts to DocumentResponse models
- Returns structured response with total count

delete_document(document_id: str) -> DeleteResponse:
- Retrieves document to get stored filename
- Deletes from database via repository
- Deletes physical file from disk
- Continues even if file deletion fails (already removed from database)
- Logs all operations
- Returns success response

Error Handling Strategy:
- Validation errors caught and re-raised with context
- Database errors bubble up from repository
- File system errors logged and handled gracefully
- Cleanup performed on partial failures

REQUEST AND RESPONSE FLOW:

Upload Request Flow:
1. Client sends POST /documents/upload with file
2. FastAPI parses multipart form data
3. Documents router receives UploadFile object
4. Router injects MongoDB database dependency
5. Router creates DocumentService instance
6. Router calls document_service.upload_document()
7. Service validates, saves, extracts, and stores
8. Service returns UploadResponse
9. Router returns HTTP 200 with response JSON
10. Client receives document ID and details

List Request Flow:
1. Client sends GET /documents/?limit=50&skip=0
2. Documents router receives query parameters
3. Router validates parameters (FastAPI automatic validation)
4. Router injects database dependency
5. Router creates DocumentService instance
6. Router calls document_service.list_documents()
7. Service calls repository.list_documents()
8. Repository queries MongoDB with projection
9. Repository returns documents and total count
10. Service converts to response models
11. Router returns HTTP 200 with JSON
12. Client receives paginated document list

Get Request Flow:
1. Client sends GET /documents/{document_id}?include_text=true
2. Documents router receives path and query parameters
3. Router injects database dependency
4. Router creates DocumentService instance
5. Router calls document_service.get_document()
6. Service calls repository.get_document_by_id()
7. Repository queries MongoDB with optional projection
8. Repository returns document data
9. Service converts to DocumentWithText model
10. Router returns HTTP 200 with JSON
11. Client receives document with full text

Delete Request Flow:
1. Client sends DELETE /documents/{document_id}
2. Documents router receives document_id path parameter
3. Router injects database dependency
4. Router creates DocumentService instance
5. Router calls document_service.delete_document()
6. Service retrieves document to get stored filename
7. Service calls repository.delete_document()
8. Repository deletes from MongoDB
9. Service deletes file from disk
10. Service returns DeleteResponse
11. Router returns HTTP 200 with confirmation
12. Client receives success message

DESIGN DECISIONS:

1. Unique Filename Generation:
   - Decision: Use UUID4 for stored filenames
   - Rationale: Prevents filename collisions, adds security through obscurity
   - Alternative considered: Hash-based names (rejected due to collision risk with truncation)

2. Text Extraction Libraries:
   - Decision: PyMuPDF for PDF, python-docx for DOCX
   - Rationale: PyMuPDF faster and more accurate than alternatives
   - Alternative considered: pdfplumber (rejected due to slower performance)

3. File Storage Location:
   - Decision: Store in configurable uploads/ directory
   - Rationale: Simple, no external dependencies, works in any environment
   - Alternative considered: Cloud storage (deferred to future optimization)

4. Repository Pattern:
   - Decision: Separate repository layer for database operations
   - Rationale: Clean Architecture, testability, potential database migration
   - Alternative considered: Direct database access in services (rejected for maintainability)

5. Projection for List:
   - Decision: Exclude documentText from list operations
   - Rationale: Performance optimization, reduce network transfer
   - Impact: Significant performance improvement for large documents

6. Error Cleanup:
   - Decision: Delete file if database insert fails
   - Rationale: Prevent orphaned files, maintain consistency
   - Implementation: try/except with cleanup in DocumentService

7. Pagination:
   - Decision: Support limit and skip parameters
   - Rationale: Handle large document collections efficiently
   - Default: limit=100, skip=0, max limit=500

8. File Validation:
   - Decision: Validate extension, MIME type, and size
   - Rationale: Security, prevent abuse, ensure compatibility
   - Approach: Fail fast with clear error messages

ASSUMPTIONS:

1. Single User Context:
   - Assumption: Authentication not yet implemented
   - Impact: userId field is optional placeholder
   - Future: Will be populated from JWT token claims

2. Local File Storage:
   - Assumption: Files stored on local disk
   - Impact: No distributed storage considerations
   - Future: May migrate to cloud storage (S3, Azure Blob)

3. English Default Language:
   - Assumption: Documents default to English
   - Impact: Language field set to "english" by default
   - Future: Auto-detection or user selection

4. Synchronous Text Extraction:
   - Assumption: Text extraction completes in reasonable time
   - Impact: Request may take several seconds for large files
   - Future: Consider background job processing for very large files

5. MongoDB Availability:
   - Assumption: MongoDB is always available
   - Impact: No retry logic or circuit breakers
   - Future: Add resilience patterns for production

6. UTF-8 Text Encoding:
   - Assumption: All document text is UTF-8 compatible
   - Impact: Non-UTF-8 text may cause issues
   - Future: Add encoding detection and conversion

KNOWN LIMITATIONS:

1. Text Extraction Quality:
   - PDF scans without OCR will extract no text
   - Complex PDF layouts may have text order issues
   - Tables and forms may not extract cleanly
   - Mitigation: Clear error message if no text found

2. File Size Constraint:
   - 25 MB limit may be restrictive for large legal documents
   - Entire file loaded into memory during processing
   - Mitigation: Configurable via MAX_FILE_SIZE_MB

3. Concurrency:
   - No locking mechanism for concurrent uploads
   - UUID filename generation prevents most conflicts
   - Database handles concurrent inserts

4. File Deletion:
   - If file deletion fails, file remains on disk
   - Database record is still deleted (inconsistency possible)
   - Mitigation: Logged for manual cleanup

5. Text Length:
   - No limit on extracted text length
   - Very large documents may cause memory issues
   - Mitigation: File size limit provides indirect control

6. Authentication:
   - No authentication currently implemented
   - All documents accessible to any client
   - Mitigation: userId placeholder ready for future auth
